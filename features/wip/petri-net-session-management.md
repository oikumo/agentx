# AgentX Petri Net Session Management

## Overview

AgentX now features advanced session state management using **Petri Nets generated by LLM**, providing a formal mathematical model for tracking user objectives throughout development sessions. This system uses Large Language Models to dynamically create workflow structures from user prompts.

**Status**: ✅ **COMPLETE** - All features implemented, tested, and validated via stdio integration tests.

**Last Updated**: May 1, 2026

## Key Features

### LLM-Based Dynamic Petri Net Generation
The system uses LLMs to automatically generate custom Petri Net structures from user prompts, rather than using fixed templates. When a user provides a query like "Debug the API 500 error", the LLM creates a complete workflow structure including:

- **Places**: Workflow states (e.g., error_reported, logs_examined, root_cause_identified)
- **Transitions**: Actions that move between states (e.g., collect_logs, analyze_patterns)
- **Edges**: Connections showing the flow between states and actions

### Session State Persistence
All generated Petri Nets are automatically saved to the session directory with timestamps:
```
.meta/sessions/petri_net_YYYYMMDD_HHMMSS_objective.json
```

### Workflow Visualization
The system provides ASCII art visualization of current Petri Net state and structure through the `status` and `petri-print` commands.

### MainController Integration ✅
All Petri Net commands are fully integrated with MainController:
- `status` - Show current Petri Net session state
- `petri-print` or `pp` - Pretty print Petri Net structure with ASCII art
- `new` - Create new sessions with Petri Net state tracking
- `help` - Shows all available commands including Petri Net commands

## Implementation Details

### Core Components

1. **LLM Petri Net Generator** (`llm_petri_net_generator.py`)
   - Uses LLM to dynamically generate custom Petri Net structures
   - Robust JSON parsing with fallback mechanisms
   - Pre-defined templates as backup for simple workflows

2. **Workflow Templates** (`workflow_templates.py`)
   - Library of common workflow patterns
   - Debug, Analysis, Implementation, Documentation workflows
   - Complex patterns with retry loops and parallel branches

3. **Objective Extraction** (`objective_extractor_llm.py`)
   - LLM-based objective extraction from prompts
   - Task classification (debug, analysis, implementation, etc.)

4. **Visualization Tools** (`petri_net_visualizer.py`)
   - ASCII art Petri Net visualization
   - Commands: `status` and `peti-print` (pp)

### How It Works

1. **User Query Processing**
   ```
   User: "Debug the API 500 error"
   ```

2. **LLM Generation**
   The LLM creates a custom workflow:
   - Places: error_reported → logs_collected → patterns_analyzed → ...
   - Transitions: collect_logs, analyze_patterns, ...
   - Initial/Final states identified
   - Reasoning provided

3. **Storage and Session Integration**
   - Petri Nets saved with timestamps in session directory
   - Integrated with MainController session management
   - Files named: `petri_net_20260501_143022_API_500_error_debug.json`

### Usage Examples

#### Generate Petri Net from Prompt
```python
from agentx.model.session.llm_petri_net_generator import generate_petri_net_from_prompt

# Generate from user prompt
manager = generate_petri_net_from_prompt("Debug the database timeout")

# Get state with metadata
state = manager.get_state()
print(f"Objective: {state.objective}")
print(f"Workflow: {state.context['workflow_type']}")
print(f"Reasoning: {state.context['llm_reasoning']}")
print(f"Enabled: {state.context['enabled_transitions']}")
```

#### Advance Through Workflow
```python
# Get enabled transitions
state = manager.get_state()
enabled = state.context.get('enabled_transitions', [])

# Advance workflow
if enabled:
    manager.advance_objective(enabled[0])
```

status # Show current Petri Net state

petri-print --help # Show visualization options
